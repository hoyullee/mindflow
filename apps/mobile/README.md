# @mindflow/mobile — Capacitor app-store shell (M7)

This package has **no application code of its own**. It is a thin Capacitor
project that wraps `apps/web`'s built PWA bundle (`apps/web/dist`) into a
native Android/iOS app so it can be submitted to the Play Store / App Store.

```
apps/mobile/
  capacitor.config.ts   # appId, appName, webDir → ../web/dist
  android/              # generated by `cap add android` — Gradle/Android Studio project (committed)
  ios/                  # generated by `cap add ios`     — Xcode project (committed)
```

`mindmap-core` and all editor/UI code stays 100% in `apps/web` — this package
only orchestrates "build the web app, then copy it into the native shell."
Nothing here is native-app-specific business logic.

## Prerequisites (to actually build/run/submit — not needed to read this repo)

| Target | Requirement |
| --- | --- |
| Android | [Android Studio](https://developer.android.com/studio) (bundles the Android SDK) + JDK 17+. Can be built/run on Linux, macOS, or Windows. |
| iOS | **macOS** + [Xcode](https://developer.apple.com/xcode/) + an Apple Developer account (for signing/submission). CocoaPods is *not* required here — Capacitor 8 generates a Swift Package Manager (`CapApp-SPM`) project instead of a Podfile. |
| Both | Node 20+, pnpm 10 (see repo root `package.json#packageManager`) |

> **This sandboxed dev environment has neither an Android SDK/emulator nor
> Xcode/macOS.** Everything below that only needs the Capacitor CLI + Node
> (scaffolding `android/`/`ios/`, `cap sync`, config/TS validation) was run
> and verified while building this feature. Actually compiling an
> APK/AAB/IPA, running an emulator/simulator, signing, and submitting to a
> store all require the real toolchain and must be done on your machine —
> see the checklist at the bottom.

## Build pipeline

`apps/web`'s Vite build output is the thing that actually runs inside the
native WebView — this package only copies it in. Order matters:

```bash
# from repo root
pnpm build:mobile
```

This runs, in order:
1. `pnpm --filter @mindflow/web build` — produces `apps/web/dist` (the same
   PWA build used for the web deploy; includes the service worker,
   manifest, icons, self-hosted font — none of that is undone here, it's
   just also embedded natively).
2. `pnpm --filter @mindflow/mobile run sync` (`cap sync`) — copies
   `apps/web/dist` into `android/app/src/main/assets/public` and
   `ios/App/App/public`, and updates each platform's plugin registrations
   (`capacitor.config.json`, `Package.swift`/Gradle plugin list) to match
   whatever Capacitor plugins are installed in `apps/web`'s
   `package.json` (see [Native bridge](#native-bridge-plugins) below).

Other scripts in `apps/mobile/package.json`:
- `pnpm --filter @mindflow/mobile run copy` — `cap copy` (web assets only,
  skips the plugin-list update step; faster iteration when you haven't
  added/removed a plugin).
- `pnpm --filter @mindflow/mobile run open:android` — opens the project in
  Android Studio (`cap open android`).
- `pnpm --filter @mindflow/mobile run open:ios` — opens the project in
  Xcode (`cap open ios`, macOS only).
- `pnpm --filter @mindflow/mobile run typecheck` — type-checks
  `capacitor.config.ts` against the root `tsconfig.base.json`.

This is intentionally **not** wired into the root Turbo `build`/`test`
pipeline (`apps/mobile/package.json` has no `build` script), so
`pnpm build`/`pnpm test`/`pnpm typecheck` at the repo root are unaffected —
`apps/mobile` is only ever touched by the explicit `pnpm build:mobile`
command or by running its scripts directly with `pnpm --filter`.

### From-scratch local run (once you have the toolchain)

```bash
pnpm install                 # repo root
pnpm build:mobile            # web build + cap sync
pnpm --filter @mindflow/mobile run open:android   # or open:ios (macOS)
# then in Android Studio / Xcode: pick a device/simulator and hit Run
```

Re-run `pnpm build:mobile` (or just `cap copy` for JS/CSS-only changes)
whenever you change anything under `apps/web/src` — the native shell only
ever sees whatever was last copied into `android/.../public` /
`ios/.../public`; it does not watch `apps/web` for changes.

## Environment variables (Supabase, etc.)

`apps/web`'s Vite build inlines `VITE_*` env vars (from `apps/web/.env.local`,
see `apps/web/.env.example`) **at build time**. If you want the shipped app
to talk to a real Supabase backend (per `server/supabase/docs/backend.md`)
rather than fall back to the local/demo adapter, set those vars *before*
running `pnpm build:mobile` — there is no runtime config step for the native
build the way there might be for a server-rendered app. Rebuild
(`pnpm build:mobile`) and re-sync whenever those values change.

## Native bridge plugins

The web app (`apps/web`) itself carries these Capacitor packages as regular
npm dependencies (not just `apps/mobile`) — its JS bundle is what actually
executes inside the WebView, so plugin *client* code has to ship with it.
Every call is gated behind `Capacitor.isNativePlatform()`
(`apps/web/src/platform/nativeBridge.ts`), which is `false` in every
ordinary browser/PWA/test run — so none of this changes web behavior, and
the web app has zero regression if a plugin is ever removed.

| Plugin | Used for | Web fallback |
| --- | --- | --- |
| `@capacitor/status-bar` | Status bar background (`#f0663f`, matches `theme-color`) + light icon style, set once at startup (`apps/web/src/platform/nativeShell.ts`). | No-op (the browser chrome color already comes from `<meta name="theme-color">` in `index.html`). |
| `@capacitor/splash-screen` | Hides the native launch splash (generated MindFlow mark on white, see [App icon / splash screen](#app-icon--splash-screen)) once the web app has mounted (`apps/web/src/platform/nativeShell.ts`); config'd `launchAutoHide: false` in `capacitor.config.ts` so it doesn't race a fixed timer. | No-op (the PWA has no native splash surface to hide — the browser/OS handles its own launch chrome). |
| `@capacitor/keyboard` | Resizes the WebView body (not the whole window) when the on-screen keyboard opens, so fixed topbars/toolbars don't get pushed off-screen while editing a node's text. | No-op (browsers already handle on-screen keyboards this way). |
| `@capacitor/filesystem` + `@capacitor/share` | Export (PNG / `.json` / `.md`, from the editor's export menu and the home doc-card menu) writes the file to the app's cache dir and opens the native Share sheet, since there's no Downloads folder / anchor-tag download inside a native WebView. | The pre-existing `URL.createObjectURL` + `<a download>` browser download (unchanged — this *is* what still runs in every browser and in all 71 existing web unit tests, since `isNativePlatform()` is false there). |
| `@capacitor/app` | Installed for future use (back-button handling, deep links, app-state resume) — not yet wired into any UI flow. | N/A. |
| `@capacitor/core` | `Capacitor.isNativePlatform()` platform detection itself. | N/A — this one small package (no native dependency) is the only one imported statically; it's what safely returns `false` everywhere above. |

Safe-area insets: `apps/web/index.html`'s
`<meta name="viewport" content="... viewport-fit=cover">` (already present
from M6) plus the app's existing CSS `env(safe-area-inset-*)` handling
covers the iPhone notch/Android gesture bar — no extra native plugin needed.

Fallback design: every native call above is wrapped in `try`/`catch` (see
`nativeShell.ts`, `nativeBridge.ts`) and logs to `console.error` rather than
throwing — a plugin failing to install/link never crashes the app or blocks
the (already-working) web code path.

## App icon / splash screen

Generated (not hand-authored, not fed through Android Studio's Image Asset
Studio / Xcode's asset catalog UI) by
[`scripts/generate-native-assets.mjs`](scripts/generate-native-assets.mjs),
a duplicate/adaptation of `apps/web/scripts/generate-icons.mjs`'s `markSvg`
(coral `#f0663f` rounded square + white "M" polyline; source of truth for the
mark itself is that web script — keep both in sync if the logo changes).
Pure SVG → `sharp` raster, no external image asset, no network fetch,
deterministic (re-running produces byte-identical output).

```bash
node apps/mobile/scripts/generate-native-assets.mjs
# or: pnpm --filter @mindflow/mobile run generate:native-assets
```

What it writes, replacing Capacitor's placeholder assets:

- **Android** (`android/app/src/main/res/`):
  - `values/ic_launcher_background.xml` — adaptive-icon background color,
    `#FFFFFF` → `#f0663f` (coral).
  - `mipmap-{mdpi,hdpi,xhdpi,xxhdpi,xxxhdpi}/ic_launcher_foreground.png`
    (108/162/216/324/432px) — white "M" only, transparent background, sized
    to fit the adaptive-icon safe zone (the coral fill comes from the
    background color above, not this layer).
  - `mipmap-{...}/ic_launcher.png` (48/72/96/144/192px, pre-API26 legacy
    icon) — full mark (coral rounded square + white M).
  - `mipmap-{...}/ic_launcher_round.png` (same sizes) — full-bleed variant
    (no baked-in corner radius; the OS applies its own circular mask).
  - `drawable*/splash.png` (11 density/orientation variants) — white
    background with the mark centered at ~32% of the shorter side; each
    file's existing width/height is read and preserved (not hardcoded), so
    it slots into whatever Capacitor's `SplashScreen` sizing expects.
- **iOS** (`ios/App/App/Assets.xcassets/`):
  - `AppIcon.appiconset/AppIcon-512@2x.png` (1024×1024) — full-bleed mark
    (iOS applies its own rounded-rect mask). `Contents.json` already points
    at this single file; unchanged.
  - `Splash.imageset/splash-2732x2732{,-1,-2}.png` (2732×2732 ×3) — same
    white-background-plus-centered-mark composition as Android.

`@capacitor/splash-screen` (added to `apps/web/package.json`, matching the
other `@capacitor/*` plugin versions) shows this native splash until the web
app explicitly hides it — see [Native bridge plugins](#native-bridge-plugins)
below and `capacitor.config.ts`'s `plugins.SplashScreen` (`launchAutoHide:
false`, `backgroundColor: '#ffffff'` to match the generated splash exactly).

Re-run the generator (and `pnpm build:mobile`) whenever the mark/brand
colors change; the script is committed so output is reproducible, not the
generation process itself.

## `cap add android` / `cap add ios` — result in this environment

Both succeeded in this sandbox (no SDK/Xcode needed for scaffolding itself):

```
$ npx cap add android   → [success] android platform added!
$ npx cap add ios       → [success] ios platform added!
```

`android/` and `ios/` are committed (Capacitor's own standard practice —
these are meant to be opened directly in Android Studio/Xcode and often
carry manual native config over time, e.g. permissions, signing config).
Their build outputs are not: Capacitor's own generated `android/.gitignore`
and `ios/.gitignore` already exclude `build/`, `.gradle/`, `Pods/`,
`DerivedData/`, and even the *copied* web bundle
(`android/app/src/main/assets/public`, `ios/App/App/public` — these are
regenerated by `cap sync`/`cap copy`, not source).

What was **not** possible here, and needs your machine:
- Actually compiling (`./gradlew assembleDebug`, or Xcode's Build), running
  on an emulator/simulator or a real device, and confirming the app launches
  and the native bridge (StatusBar color, Share sheet on export) behaves as
  expected.
- Code signing (Android keystore / Play App Signing; iOS provisioning
  profile + certificate) — this needs your own Google Play / Apple Developer
  accounts and is store-submission-time, not build-time, in this repo.

## Store submission checklist (do this on your machine)

**Android (Play Store)**
- [ ] Open in Android Studio (`pnpm --filter @mindflow/mobile run
      open:android`), let Gradle sync, run on an emulator/device to confirm
      it launches and looks right (status bar color, keyboard resize,
      export → Share sheet).
- [ ] Set `versionCode`/`versionName` in `android/app/build.gradle`.
- [ ] Generate/reuse a release keystore; configure signing config (do **not**
      commit the keystore or its passwords — keep them outside this repo,
      e.g. in your CI secret store or local `~/.gradle/gradle.properties`).
- [ ] Build a release AAB: `./gradlew bundleRelease` (from `android/`).
- [ ] Launcher icons/splash are already generated (see [App icon / splash
      screen](#app-icon--splash-screen)) — re-run the generator first if the
      mark/brand color has changed since the last release.
- [ ] Create the app listing in Play Console (screenshots, privacy policy —
      required since the app can persist/export user-authored content;
      MindFlow's own privacy posture depends on which backend — Local vs
      Supabase — you shipped with, see `server/supabase/docs/backend.md`).
- [ ] Upload the AAB, complete the content rating questionnaire, submit for
      review.

**iOS (App Store)**
- [ ] On macOS: open in Xcode (`pnpm --filter @mindflow/mobile run
      open:ios`), let Swift Package resolution finish, run on a simulator to
      confirm the same behaviors as above.
- [ ] Set your Team/Signing (Apple Developer Program membership required)
      under the App target's Signing & Capabilities.
- [ ] Bump `CFBundleShortVersionString`/`CFBundleVersion` in `Info.plist`.
- [ ] App icon/splash are already generated (see [App icon / splash
      screen](#app-icon--splash-screen)) — re-run the generator first if the
      mark/brand color has changed since the last release.
- [ ] Archive (Product → Archive) and upload via Xcode Organizer or
      `xcodebuild`/`altool`/`xcrun notarytool` from the command line.
- [ ] Create the App Store Connect listing (screenshots for each supported
      device size, privacy nutrition label, App Review notes) and submit.

**Both**
- [ ] Confirm `apps/web/.env.local` had the Supabase vars you intend for
      production *before* running `pnpm build:mobile` for the release build
      (env is inlined at web-build time — see above).
- [ ] Re-run `pnpm build:mobile` right before each release build so the
      native shell embeds the latest web bundle.

## Known limitations (of this pass, honestly)

- No CI/emulator verification of the native build — only `cap add`,
  `cap sync`, `capacitor.config.ts` validity, and `apps/mobile`'s
  `tsc --noEmit` were exercised, because this environment has no Android
  SDK/Xcode.
- `@capacitor/app`'s back-button/deep-link/resume hooks are installed but
  not yet wired into any screen.
- Push notifications, biometric auth, and other deeper native features are
  out of scope for this pass.
